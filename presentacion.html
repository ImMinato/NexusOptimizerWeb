Presentación del proyecto

Diseño, desarrollo e implementación de un sistema de optimización de rendimiento

NexusOptimizer es una aplicación de escritorio para Windows desarrollada en C# sobre el framework WPF, orientada al diagnóstico, mantenimiento y optimización de rendimiento de equipos de cómputo. El sistema está organizado en más de 27 módulos independientes que cubren seis áreas funcionales: hardware y energía, disco y almacenamiento, red y privacidad, mantenimiento del sistema, diagnóstico mediante benchmarking, e infraestructura de soporte.

El código fuente, de aproximadamente 11.000 líneas distribuidas en 35 archivos, fue desarrollado, probado y documentado en su totalidad por el equipo a lo largo de casi un año de trabajo, presentado para la Feria de Ciencias y Tecnología 2026.

De un script de Batch a una aplicación completa

NexusOptimizer no nació desde cero

El proyecto es la evolución de un script de línea de comandos (OpV6_4.bat, "ByteFix Edition") desarrollado previamente por el mismo equipo desde mediados de 2025, del cual se conservó y amplió la lógica de optimización ya validada, agregando una interfaz gráfica completa, persistencia de datos estructurada, capacidades de diagnóstico en tiempo real y un sistema de perfiles que coordina múltiples optimizaciones con una sola acción del usuario.

Una verificación de trazabilidad confirmó coincidencia exacta en 13 de los parámetros más relevantes entre ambas versiones —incluyendo el GUID del plan de energía "Máximo rendimiento" y el valor de NetworkThrottlingIndex— demostrando que la lógica de optimización no se reescribió desde cero, sino que se migró y extendió sobre una base ya validada empíricamente.

Objetivos del proyecto

Objetivo general y objetivos específicos planteados al inicio

Objetivo general: diseñar, desarrollar e implementar una aplicación de escritorio para Windows que permita diagnosticar, optimizar y mantener el rendimiento de un equipo de cómputo de forma segura, reversible y comprensible para el usuario final.

Migrar y ampliar

Migrar la lógica validada del script Batch hacia una arquitectura más robusta, con interfaz gráfica y manejo estructurado de errores.

Arquitectura modular

Cada función del sistema como una unidad de código independiente, probable y mantenible por separado.

Benchmark con historial

Medir el rendimiento antes y después de optimizar, con persistencia de un historial de sesiones.

Mecanismos de seguridad

Puntos de restauración y copias del registro previos a cualquier cambio, para garantizar reversibilidad total.

Perfiles de optimización

Agrupar múltiples módulos según un objetivo de uso concreto, reduciendo la complejidad para el usuario.

Base de conocimiento

Buenas prácticas de mantenimiento de hardware y software, organizadas por categoría e importancia.

Arquitectura del sistema

Cuatro capas conceptuales, con separación clara de responsabilidades

Capa de presentación: interfaz gráfica WPF/XAML, 7 ventanas, 16 pestañas
Capa de lógica: 27 clases estáticas, una por dominio de optimización o diagnóstico
Servicios transversales: Logger, SettingsManager, NotificationService, RestorePointHelper, UpdateChecker
Capa de sistema: Registro de Windows, WMI, PerformanceCounters y P/Invoke nativo

Marco teórico

Conceptos centrales detrás de las optimizaciones

Registro de Windows

Base de datos jerárquica organizada en "colmenas" (HKLM, HKCU) donde se almacenan los valores que controlan el comportamiento de bajo nivel del sistema.

WMI

Infraestructura estándar de Microsoft para consultar hardware y sistema mediante un lenguaje similar a SQL (WQL), usada para identificar CPU, GPU, discos y temperatura.

Performance Counters

Mecanismo de .NET para monitorear uso de CPU, RAM y disco en tiempo real sin acceder directamente a estructuras del kernel.

Planes de energía

Configuraciones identificadas por GUID que controlan frecuencias de CPU y comportamiento de ahorro de energía, incluyendo el plan oculto "Ultimate Performance".

MMCSS

Servicio de Windows que prioriza CPU para procesos de audio/video, evitando cortes de sonido mediante el parámetro SystemResponsiveness.

Algoritmo de Nagle

Técnica TCP que agrupa paquetes pequeños antes de enviarlos; desactivarla reduce la latencia a costa de mayor overhead de red.

Pruebas sobre hardware real

Validado en un equipo real del laboratorio, no en máquinas virtuales

Todas las pruebas funcionales se realizaron sobre un equipo con procesador AMD Athlon 320GE (gráficos integrados Radeon Vega), placa de video NVIDIA GeForce GT 710, 7,9 GB de RAM y una unidad SSD de 222,8 GB, bajo Windows 10 IoT Enterprise LTSC 2021. Se priorizó hardware real porque buena parte de la lógica del sistema —lectura de sensores de temperatura, contadores de rendimiento, identificación de hardware— depende de controladores y datos no siempre representativos en un entorno virtualizado.

Las pruebas confirmaron el funcionamiento correcto de la detección automática de CPU y GPU, el cálculo de espacio libre en disco, el monitoreo en tiempo real de CPU y RAM, y el mecanismo de respaldo en cascada del módulo de temperatura.
